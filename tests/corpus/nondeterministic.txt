# ROOT CAUSE (identified 2026-07-07): `expand_cte_entity`
# (src/render_plan/select_builder.rs:1590) builds the SELECT list for a bare
# node/relationship variable (RETURN n / RETURN r / collect(n) / nodes(p),
# NOT a `.prop` access) that resolves to a CTE — i.e. any such reference
# downstream of a WITH barrier, or bound to a VLP/pattern-combinations CTE.
# It sources properties via `schema.get_node_properties`/
# `get_relationship_properties` (graph_schema.rs:1408-1443, both iterate a
# `HashMap<String, PropertyValue>` with no `.sort()`) and, for denormalized
# nodes, an unsorted merge over `from_properties`/`to_properties`
# (`HashMap<String,String>`, select_builder.rs:1687/1698). The identical
# `get_relationship_properties` call is already sorted at two OTHER call
# sites in the same file (select_builder.rs:2454-2460/2527-2533, citing
# issue #464) — `expand_cte_entity` was evidently missed by that fix. A
# second, narrower source: the denormalized VLP CTE builder
# (src/render_plan/cte_extraction.rs:3705-3795) iterates
# `from_properties`/`to_properties` directly (lines ~3751/3775) into the
# CTE's fixed column order, no sort — triggers on `*`/`*1..N` patterns where
# BOTH endpoints are denormalized/virtual nodes. This is a genuine,
# pre-existing rendering bug (not something P0.6 fixes — test-only slice),
# already the same class as the `#464`-tagged nondeterminism documented in
# `sql_golden_tests.rs`.
#
# This list was assembled empirically (150+ process invocations of the
# sweep, unioning every diff between them) rather than by exhaustively
# enumerating every query touching the code paths above — a regex-based
# over-approximation of "WITH ... RETURN <bare node/rel var>" or "denorm
# schema + VLP `*`" flags ~90 corpus entries as *structurally at risk*, most
# of which did not flap in 150+ observed runs (the bug requires actual
# 2+-key ordering ambiguity in the specific unsorted map involved, not just
# syntactic shape) — so more entries in this broader risk class MAY start
# flapping under different hash-seed randomization in future runs. Any CI
# flake matching this signature (a corpus_sweep MISMATCH where two runs
# differ only in property/column ORDER within a WITH-crossing whole-entity
# or denormalized-VLP query) should be triaged as "add to this file", not
# "regression" — see corpus_sweep.rs's module docs.
